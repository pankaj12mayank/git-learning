# 🔐 1️⃣ Branch Protection Rules (VERY IMPORTANT)

This is PM-level control.

Without protection:
Anyone can push directly to `main`.

With protection:
✔ PR mandatory
✔ Approval required
✔ CI must pass
✔ Force push disabled

---

## 🧪 Configure Branch Protection (Do This)

1. Go to your repository on GitHub
2. Click **Settings**
3. Click **Branches**
4. Click **Add branch protection rule**

In Branch name pattern:

```
main
```

Enable:

✔ Require a pull request before merging
✔ Require approvals (set 1)
✔ Dismiss stale approvals when new commits are pushed
✔ Require status checks to pass (optional for now)
✔ Do not allow force pushes

Click **Save changes**

# 📊 3️⃣ GitHub Projects (Kanban Board)

Go to:
👉 Projects
👉 New Project
👉 Board template

You’ll see columns:

* To Do
* In Progress
* Done

Now link your Issue to board.

Move card across columns.

---

# 🧠 Real-World Flow

1. Create Issue
2. Developer creates branch:

   ```
   feature/payment-123
   ```
3. Developer references issue in PR:

   ```
   Fixes #123
   ```
4. When PR merges → Issue auto closes

This creates full traceability.

---

# 🎯 PM Governance Framework Using GitHub

As PM you should ensure:

✔ All work linked to Issues
✔ No direct push to main
✔ All merges via PR
✔ Approvals required
✔ Proper branch naming convention
✔ Sprint board maintained

---

# 📌 Branch Naming Best Practice

```
feature/<feature-name>
bugfix/<bug-id>
hotfix/<production-fix>
release/<version>
```
